# Claude Working Guide For This Repository

This repository is the Nirnex CLI monorepo.

It is not a normal application repo. Most changes here affect one of four contracts:

1. CLI surface and setup/remove flows
2. Claude hook runtime and zero-trust validation
3. Indexing / parser / scope classification
4. Ledger, report, and replay data integrity

## Repository Shape

- `packages/cli`: user-facing CLI commands and Claude hook runtime
- `packages/core`: deterministic logic, SQLite access, ECO, ledger, reporting, scope, reproducibility
- `packages/parser`: tree-sitter based parsing and compatibility checks
- `packages/hooks`: lightweight package, currently thin
- `site`: Docusaurus docs site with its own package.json and build pipeline
- `tests`: regression-heavy Vitest suite, including many sprint bugfix tests

## Important Local Reality

Do not "turn Nirnex on" for this repository unless explicitly asked.

This repo currently has:
- `.claude/settings.json` with `"disableAllHooks": true`
- no `nirnex.config.json`

That is deliberate. The project should not accidentally govern its own development while you are editing the implementation.

## Naming And Compatibility Constraints

Do not casually normalize names.

There is intentional mixed naming across the codebase:
- product and packages: `nirnex`
- repo folder / historical runtime names: `AIDOS`, `.aidos.db`, `.aidos-ledger.db`

Those names are part of the runtime and test surface. Do not rename them as cleanup.

## Toolchain

Root workspace:
- Node `>=24`
- npm workspaces
- build: `npm run build`
- typecheck: `npm run typecheck`
- lint: `npm run lint`
- tests: `npm run tests`

Docs site:
- separate package in `site/`
- Node `>=20`
- verify with `cd site && npm run typecheck && npm run build`

Root lint intentionally ignores `site/**`. Do not assume root validation covers docs.

## Dist Import Constraint

A lot of source files in `packages/cli/src` import built outputs such as:
- `@nirnex/core/dist/...`
- `@nirnex/parser/dist/...`

That means API changes often require a root build before typecheck/tests are meaningful.

If you change exports, paths, or shared types:
1. run `npm run build`
2. then run the targeted tests
3. then run broader verification if the change is cross-cutting

## Runtime Contract Rules

When touching hook/runtime code, preserve these invariants:

- Hook scripts generated by setup must use direct-node-entry, not `env node`
- `validate` must make decisions from trace/evidence, not model prose
- `ledger` is canonical
- `hook-events.jsonl` is observational, not authoritative
- envelope state is ephemeral and task-scoped
- `finalized_at` idempotency must work for both allow and block paths
- verification matching must stay centralized around `isBashVerificationCommand`
- reason-code literals must stay aligned across CLI and core runtime modules

Primary files:
- `packages/cli/src/runtime/*.ts`
- `packages/cli/src/commands/setup.ts`
- `packages/cli/src/commands/remove.ts`
- `packages/cli/src/commands/doctor.ts`
- `packages/core/src/runtime/*.ts`

## Index / Parser Rules

When touching indexing or parsing, preserve these invariants:

- `packages/core/src/db.ts` schema versioning and `.aidos.db` semantics
- `packages/core/src/schema.sql` must still be copied into `packages/core/dist/` by build
- parse diagnostics stages in parser are part of debugging behavior
- parser compatibility matrix and smoke tests must match dependency expectations
- scope classification output must remain explainable through `--explain-scope`

Primary files:
- `packages/cli/src/commands/index.ts`
- `packages/parser/src/index.ts`
- `packages/parser/src/compatibility.ts`
- `packages/core/src/db.ts`
- `packages/core/src/scope/*`

## Docs And Surface Sync

If you change CLI behavior, hook artifacts, runtime files, or setup/remove semantics, update the docs in the same change.

At minimum check:
- `README.md`
- `CODE.md`
- relevant `site/docs/**`

If you change site content or components, verify inside `site/` separately.

## XState / ast-grep Note

`nirnex query` shells out to `npx sg scan --json`.
The repo currently ships ast-grep rules in `rules/` for XState-related patterns.

If you change query behavior around state machines or pattern detection, review:
- `rules/xstate-create.yml`
- `rules/xstate-guard.yml`
- `rules/xstate-transition.yml`

## Minimum Verification Matrix

Use the smallest meaningful set.

Setup/remove/doctor changes:
- `npx vitest run tests/setup.test.ts tests/remove.test.ts tests/doctor.test.ts`

Hook runtime / validate / audit changes:
- `npx vitest run tests/hook-audit.test.ts tests/evidence-integrity.test.ts tests/store-hierarchy.test.ts tests/stop-hook-idempotency.test.ts tests/stdin-transport.test.ts`

Index / parser / scope changes:
- `npx vitest run tests/parser-compat.test.ts tests/scope.test.ts`

Cross-cutting CLI/core changes:
- `npm run build`
- `npm run typecheck`
- `npm run tests`

## Change Style For This Repo

Prefer surgical fixes over architecture churn.

Do not:
- rewrite package boundaries casually
- replace append-only flows with mutable ones
- collapse separate stores into one abstraction
- rename reason codes or runtime files without a migration plan
- "clean up" mixed AIDOS/Nirnex naming unless the task is explicitly a migration

Do:
- keep the contract explicit
- keep evidence machine-verifiable
- add or update tests with every behavioral fix
- document user-visible CLI/setup/runtime changes immediately
- TDD development
- Always run test cases
- Always run lint
- Update the documents
