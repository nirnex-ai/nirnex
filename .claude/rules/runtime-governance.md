# Runtime Governance Rule

Use this rule whenever you touch any of:
- `packages/cli/src/runtime/*`
- `packages/cli/src/commands/setup.ts`
- `packages/cli/src/commands/remove.ts`
- `packages/cli/src/commands/doctor.ts`
- `packages/core/src/runtime/*`

## Non-negotiable invariants

- Hook launchers generated by setup must use direct-node-entry
- `validate` decisions must come from machine evidence, not prose summaries
- `ledger` is canonical
- `hook-events.jsonl` is an append-only audit trail, not the source of truth
- envelope files are task-scoped execution state
- verification is matched by command evidence, not by text claiming tests passed
- once verification ran, later edits must remain blocked by Rule 3
- stop-hook idempotency must prevent duplicate final outcomes
- block-path idempotency matters as much as allow-path idempotency

## Synchronization requirements

If you add or rename a reason code or violation code, keep literals aligned across:
- `packages/cli/src/runtime/types.ts`
- `packages/core/src/runtime/store-hierarchy.ts`
- `packages/core/src/runtime/evidence-integrity.ts`

If you change hook event shapes, update:
- `packages/cli/src/runtime/types.ts`
- any code reading/writing hook events
- hook-log output expectations
- affected tests

If you change setup hook templates, also update:
- remove/doctor compatibility logic
- README/CODE docs that describe generated hooks

## Verification

At minimum run:
- `npx vitest run tests/hook-audit.test.ts tests/evidence-integrity.test.ts tests/store-hierarchy.test.ts tests/stop-hook-idempotency.test.ts tests/stdin-transport.test.ts`

If setup/remove/doctor is touched, also run:
- `npx vitest run tests/setup.test.ts tests/remove.test.ts tests/doctor.test.ts`

For cross-cutting runtime edits, finish with:
- `npm run build`
- `npm run typecheck`
